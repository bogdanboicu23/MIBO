# Streaming & Live Runtime

The live assistant path in MIBO is built on **Server-Sent Events**. The client does not wait for one final JSON document at the end of a normal HTTP request. Instead, it receives structured progress and final payload events over a streaming response.

## Streaming boundary

The browser talks to:

```http
POST /api/v1/chat
```

`ConversationService` then opens an upstream streamed call to:

```http
POST /chat
```

on `MIBO.LangChainService`.

## Response headers

The downstream browser response is returned with:

| Header | Purpose |
| --- | --- |
| `Content-Type: text/event-stream` | declares SSE transport |
| `Cache-Control: no-cache` | avoids aggressive buffering |
| `X-Accel-Buffering: no` | reverse-proxy buffering compatibility |
| `X-Conversation-Id` | exposes the resolved conversation ID |

## Event types used in the runtime

| Event type | Producer | Meaning |
| --- | --- | --- |
| `session` | ConversationService or LangChainService | communicates the resolved session or conversation ID |
| `status` | LangChainService | human-readable progress update from a graph phase |
| `chunk` | LangChainService | partial fragment of the final structured JSON payload |
| `done` | LangChainService | complete final response serialized as JSON text |
| `error` | optional | failure information when available |

## Event format

The wire format is standard SSE:

```text
data: {"type":"status","content":"Understanding your request..."}

data: {"type":"chunk","content":"{\"text\":\"..."}

data: {"type":"done","content":"{\"text\":\"Final answer\",\"components\":[]}"}
```

## Upstream versus downstream behavior

### Upstream: LangChainService

The AI service emits:

- `session` only when it had to create a session ID itself
- `status` from `intent_agent`, `planner_agent`, and `executor_agent`
- `chunk` as the composer streams partial JSON
- `done` once the final structured payload is available

### Downstream: ConversationService

The outer service mostly forwards the stream unchanged.

It adds behavior in two places:

- it can emit its own synthetic `session` event when it created the conversation before calling the agent
- it inspects the `done` event so it can persist `assistantPayload` and `assistantText`

## Why `chunk` exists even though `done` is authoritative

The final payload is authoritative, but `chunk` has two practical purposes:

1. it allows progressive downstream delivery while the composer is still generating
2. it gives the client a recovery path if the final payload arrives as streamed fragments before the terminal `done`

In the current client implementation, `done` is still the decisive event used to finalize the assistant message and build `ui.v1`.

## Client-side parsing model

The React client:

1. opens `fetch()` against `/api/v1/chat`
2. reads the `ReadableStream`
3. appends incoming bytes into a string buffer
4. splits events on `\n\n`
5. extracts `data:` lines
6. parses the event JSON

The runtime behavior per event is:

- `status`: update the temporary assistant status text
- `chunk`: append to a local chunk buffer
- `done`: parse the final payload as `AgentFinalResponse`, build `ui.v1`, and finalize the assistant message

## Structured response semantics

<Callout type="info" title="The stream carries structured state, not only text">
  The assistant is not designed as a token-only text chatbot. The streamed output is a transport for a structured final response that may include components, reusable data sources, reusable actions, and subscriptions.
</Callout>

## Persistence hook in ConversationService

`ConversationService` uses SSE parsing for one important side effect: it watches for the `done` event and attempts to parse the embedded JSON payload.

If parsing succeeds:

- the whole object is stored as `assistantPayload`
- the top-level `text` field is extracted as the assistant's message text

If parsing fails:

- the raw content is still persisted as the assistant text fallback

## Failure modes

### Upstream non-success response

If `LangChainService` returns a non-success HTTP status, `ConversationService` stops the relay and propagates the error downstream.

### Stream closes before `done`

The client treats this as an error because the structured final payload is missing.

### Malformed model output

The AI service attempts to recover malformed JSON during composition before emitting `done`, which reduces client-side parsing failures.

## Practical mental model

Treat the stream as a phased conversation protocol:

- `session` establishes identity for the conversation
- `status` reflects internal AI phases
- `chunk` reflects progressive assembly of the final JSON
- `done` closes the loop with the actual assistant contract the client can render
