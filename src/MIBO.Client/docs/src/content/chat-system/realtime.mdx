# Real-Time Features

MIBO uses two real-time communication channels: **Server-Sent Events (SSE)** for streaming AI responses and **SignalR** for UI updates.

<SectionDivider />

<SectionHeader title="SSE Streaming" description="Token-by-token delivery of AI responses for a typewriter effect" />

The `/api/conversations/stream` endpoint delivers AI responses token-by-token for a typewriter effect.

<Callout type="info" title="Streaming Endpoint">
  The SSE stream is accessed via `POST /api/conversations/stream` with a JSON body and Bearer token authentication.
</Callout>

### Client Implementation

```typescript
const response = await fetch("/api/conversations/stream", {
    method: "POST",
    headers: {
        "Content-Type": "application/json",
        "Authorization": `Bearer ${token}`
    },
    body: JSON.stringify({ message: "What's my balance?" })
});

const reader = response.body?.getReader();
const decoder = new TextDecoder();

while (true) {
    const { done, value } = await reader.read();
    if (done) break;

    const chunk = decoder.decode(value);
    // Parse SSE data lines
    const lines = chunk.split("\n");
    for (const line of lines) {
        if (line.startsWith("data: ")) {
            const data = JSON.parse(line.slice(6));
            if (data.done) break;
            appendToken(data);
        }
    }
}
```

### SSE Event Format

```
data: {"token": "Here"}

data: {"token": " are"}

data: {"token": " your"}

data: {"done": true}
```

<SectionHeader title="SignalR UI Hub" description="Real-time bidirectional communication for live UI patches" />

The SignalR hub at `/hubs/ui` enables real-time UI patches from the server.

### Connection Setup

```typescript
import { HubConnectionBuilder } from "@microsoft/signalr";

const connection = new HubConnectionBuilder()
    .withUrl("/hubs/ui")
    .withAutomaticReconnect()
    .build();

await connection.start();
```

### Methods

| Method | Direction | Description |
|--------|-----------|-------------|
| `JoinConversation(id)` | Client → Server | Subscribe to a conversation's updates |
| `LeaveConversation(id)` | Client → Server | Unsubscribe from updates |
| `UiPatch(patch)` | Server → Client | Receive UI update patches |

### Receiving UI Patches

When the server processes an action or completes a tool execution, it sends UI patches to all clients in the conversation group:

```typescript
connection.on("UiPatch", (patch) => {
    // Apply the UI update to the current view
    applyUiPatch(patch);
});
```

<Callout type="tip" title="Group Management">
  Conversations are managed as SignalR groups using the pattern `conversation:{conversationId}`. When a client joins, it receives all subsequent UI updates for that conversation.
</Callout>

<SectionHeader title="NATS JetStream Events" description="Backend event streaming for decoupled service communication" />

Backend events flow through NATS JetStream with the stream name `MIBO_EVENTS`.

### Event Envelope

```json
{
    "schema": "event.v1",
    "subject": "finance.expense_created",
    "eventId": "guid-here",
    "correlationId": "corr-123",
    "conversationId": "conv-456",
    "userId": "user-789",
    "occurredAtUtc": "2024-01-15T10:30:00Z",
    "payload": {
        "expenseId": 42,
        "amount": 45.00,
        "category": "groceries"
    }
}
```

### Subscribed Subjects

| Subject Pattern | Description |
|----------------|-------------|
| `finance.*` | All financial events |
| `shop.*` | Shopping-related events |

<Callout type="info" title="Durable Consumer">
  The Conversation Service uses a durable NATS consumer (`conversationservice_refresh`) to ensure events are not lost during service restarts.
</Callout>
