# Deployment

MIBO currently has two deployment stories:

1. **AppHost** for local orchestration
2. **Kubernetes plus GitHub Actions** for the deployed test environment represented in this repository

## Current deployment split

| Area | Current source of truth |
| --- | --- |
| Local orchestration | `src/MIBO.AppHost/AppHost.cs` |
| Backend service charts | `deploy/helm/*` |
| Backend environment values | `deploy/environments/test/*` |
| Frontend/docs/status manifests | `k8s/apps/*` |
| Ingress controller | `k8s/traefik/*` |
| CI/CD workflows | `.github/workflows/*` |

## What is deployed from this repository

### Backends via Helm

- `api-gateway`
- `conversation-service`
- `action-service`
- `langchain-service`
- `identity-service`

### Frontend and auxiliary apps via raw manifests

- `mibo-client`
- `mibo-docs`
- `mibo-status`

### Cluster resources tracked in the repository

- Traefik ingress resources
- frontend application manifests
- supporting platform manifests under `k8s/`

## What is environment-dependent

Some backing services are configured through secrets or external connection strings rather than fully provisioned by the Kubernetes manifests in this repository:

- PostgreSQL
- MongoDB
- RabbitMQ

Those are explicit in AppHost for local development, but the Kubernetes side relies on environment values and secrets rather than one unified in-repo cluster bootstrap.
