# Observability

MIBO uses a full observability stack for monitoring, logging, and metrics collection across all microservices.

<SectionDivider />

<SectionHeader title="Stack Overview" description="Telemetry flows from services through OpenTelemetry collectors to Prometheus and Loki, visualized in Grafana." />

```
   MIBO Services
       │
       │  OTLP (gRPC + HTTP)
       ▼
┌──────────────────┐
│  OpenTelemetry   │  Collectors (DaemonSet + Deployment)
│  Collector       │
└────────┬─────────┘
         │
    ┌────┴─────┐
    ▼          ▼
┌────────┐ ┌──────┐
│Promethe│ │ Loki │  Log aggregation
│  us    │ │      │
└────┬───┘ └──┬───┘
     │        │
     └────┬───┘
          ▼
    ┌──────────┐
    │ Grafana  │  Dashboards & visualization
    └──────────┘
```

<SectionHeader title="OpenTelemetry Collectors" description="Two collector modes handle different telemetry data." />

### DaemonSet Collector

Runs on every node to collect node-level telemetry:

```yaml
# deploy/opentelemetry/values.yaml
mode: daemonset
presets:
  kubernetesAttributes: true   # Add k8s metadata to all signals
  kubeletMetrics: true          # Collect kubelet metrics per node
  logsCollection: true          # Collect container file logs
```

### Deployment Collector

A single replica collecting cluster-wide metrics:

```yaml
# deploy/opentelemetry/receiver.yaml
mode: deployment
replicas: 1
presets:
  clusterMetrics: true          # k8s cluster-level metrics
  kubernetesEvents: true        # k8s events as logs
```

<SectionDivider />

<SectionHeader title="Collector Pipeline" description="The OpenTelemetry collector pipeline configuration for receiving, processing, and exporting telemetry." />

```yaml
# deploy/opentelemetry/collector.yaml
image: otel/opentelemetry-collector:0.145.0
config:
  receivers:
    otlp:
      protocols:
        grpc: {}    # gRPC receiver for traces/metrics
        http: {}    # HTTP receiver for traces/metrics
  processors:
    batch: {}
    memory_limiter:
      limit_percentage: 75
      spike_limit_percentage: 15
  exporters:
    prometheus:
      endpoint: "0.0.0.0:8889"
  pipelines:
    metrics:
      receivers: [otlp]
      processors: [memory_limiter, batch]
      exporters: [prometheus]
```

<Callout type="tip" title="OTLP Protocol">
  Services send telemetry data to the OpenTelemetry collector via OTLP (gRPC or HTTP). The collector then processes, batches, and exports the data to the appropriate backend (Prometheus for metrics, Loki for logs).
</Callout>

<SectionDivider />

<SectionHeader title="What's Monitored" description="Signals collected across both DaemonSet and Deployment collectors." />

| Signal | Source | Collector |
|--------|--------|-----------|
| Container logs | Node filesystem | DaemonSet |
| Kubelet metrics | Each node's kubelet | DaemonSet |
| k8s attributes | Pod/node metadata | DaemonSet |
| Cluster metrics | k8s API server | Deployment |
| k8s events | k8s API server | Deployment |
| Application metrics | OTLP from services | Both |

<SectionDivider />

<SectionHeader title="Memory Protection" description="The collector includes a memory limiter processor to prevent OOM kills." />

```yaml
memory_limiter:
  limit_percentage: 75       # Hard limit at 75% of available memory
  spike_limit_percentage: 15  # Allow 15% spike above limit
```

<Callout type="warning" title="Pipeline Ordering">
  The memory limiter is applied **before** the batch processor in the pipeline to drop data early if memory pressure is high. This prevents the collector from being OOM-killed under heavy load.
</Callout>
