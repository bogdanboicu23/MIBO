# Docker & Images

Every major service and frontend in MIBO has a Docker build path, and GitHub Actions uses those Dockerfiles to publish images.

## Active Docker build targets

| Area | Dockerfile path |
| --- | --- |
| API Gateway | `src/MIBO.ApiGateway/Dockerfile` |
| Identity Service | `src/MIBO.IdentityService/Dockerfile` |
| Conversation Service | `src/MIBO.ConversationService/Dockerfile` |
| Action Service | `src/MIBO.ActionService/Dockerfile` |
| LangChain Service | `src/MIBO.LangChainService/Dockerfile` |
| Client | `src/MIBO.Client/client/Dockerfile` |
| Docs | `src/MIBO.Client/docs/Dockerfile` |
| Status | `src/MIBO.Client/status/Dockerfile` |

## Local recommendation

For local multi-service runtime, prefer **AppHost** because it:

- wires the service graph used by the application
- provisions the supporting infrastructure required by the services
- injects the runtime configuration expected by the platform

## CI/CD image publishing

The GitHub Actions workflows build and publish images to:

```text
registry.digitalocean.com/mibo/
```

Typical tags include:

- `test-latest`
- `test-<sha>`
- `test-<run_id>-<run_attempt>`
- `pr-<number>-<sha>` for pull-request builds in some workflows

## LangChain image specifics

The Python agent image:

- uses `python:3.11-slim`
- installs dependencies from `requirements.txt`
- runs `uvicorn main:app`
- bundles the agent code and config into the image
