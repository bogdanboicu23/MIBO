# Infrastructure

MIBO relies on a small set of core infrastructure services that are configured differently between local development and the deployed environment.

## Infrastructure by concern

| Service | Local AppHost | Deployed environment | Current purpose |
| --- | --- | --- | --- |
| Redis | yes | configured through environment or cluster resources | Spotify tokens and finance connection state |
| MongoDB | yes | configured through secrets and connection strings | conversations, messages, UI instances, monitoring data |
| PostgreSQL | yes | configured through secrets and migration jobs | IdentityService relational storage |
| RabbitMQ | yes | configured through environment values | ActionService retry flow |

## In-repo infrastructure assets

The repository contains infrastructure assets in three main places:

- `src/MIBO.AppHost/AppHost.cs` for local orchestration
- `deploy/migrations/*` for Identity database schema changes
- `k8s/traefik/*` and selected resources under `k8s/apps/*` for cluster-side support

## Environment model

The deployed test environment relies on environment variables, secrets, and Helm values rather than a single in-repo full-stack bootstrap. In practice that means:

- AppHost is the clearest reference for local development
- Helm values define the runtime wiring for backend services
- GitHub Actions applies the frontend and ingress manifests used by the cluster
