# Chat

`ConversationService` exposes the chat endpoint used by the React client through the gateway:

| Method | Route | Purpose |
| --- | --- | --- |
| `POST` | `/api/v1/chat` | main contract used by the React client |

The endpoint returns `text/event-stream`.

## Main contract: `POST /api/v1/chat`

Request:

```json
{
  "conversationId": "optional-existing-id",
  "prompt": "Show me my spending this month"
}
```

`message` is also accepted as an alternative field for compatibility.

### Real behavior

- if `conversationId` is missing, the service creates a new conversation
- the user message is persisted immediately in Mongo
- the service forwards the request to `LangChainService`
- the upstream stream is relayed to the client as SSE
- the assistant result is persisted after a successful `done` event

### Upstream request to the AI service

`ConversationService` forwards the normalized request to `LangChainService` as:

```json
{
  "session_id": "resolved-conversation-id",
  "message": "Show me my spending this month",
  "user_id": "resolved-user-id"
}
```

### Persistence side effects

This endpoint is not only a transport endpoint. It also mutates conversation state:

- appends a user message before the upstream AI call
- updates `lastMessagePreview`, `lastMessageAt`, and `messageCount`
- auto-titles the conversation from the first user message when needed
- appends an assistant message with `assistantPayload` after the `done` event arrives

## Response headers

| Header | When present | Purpose |
| --- | --- | --- |
| `Content-Type: text/event-stream` | always | SSE response stream |
| `X-Conversation-Id` | `/api/v1/chat` | exposes the resolved conversation ID |
| `Cache-Control: no-cache` | always | reduces aggressive buffering |
| `X-Accel-Buffering: no` | always | reverse-proxy compatibility |

## Common errors

| Case | Status |
| --- | --- |
| prompt missing or blank | `400` |
| conversation does not exist for this user | `404` |
| upstream agent failure | propagated upstream status |
