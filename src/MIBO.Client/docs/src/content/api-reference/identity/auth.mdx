# Identity & Auth

All endpoints below are publicly exposed through the gateway under `/api/auth`.

## Core endpoints

| Method | Route | Auth | Purpose |
| --- | --- | --- | --- |
| `POST` | `/api/auth/login` | no | authenticate with email and password |
| `POST` | `/api/auth/register` | no | create a new user |
| `POST` | `/api/auth/refresh-token` | no, but cookie required | issue a new access token |
| `GET` | `/api/auth/me` | yes | return current user information and roles |
| `GET` | `/api/auth/test` | no | simple health/info endpoint |
| `GET` | `/api/auth/test-auth` | yes | authenticated claim echo |

## Login

```http
POST /api/auth/login
Content-Type: application/json
```

```json
{
  "email": "user@example.com",
  "password": "Test1234",
  "turnstileToken": "optional-or-required-depending-on-config"
}
```

Successful response:

```json
{
  "jwtToken": "eyJ..."
}
```

### Real behavior

- if `Turnstile:Enabled=true`, a captcha token is required
- the refresh token is not returned in the response body
- the refresh token is written to the `refreshToken` HTTP-only cookie
- cookie settings depend on HTTPS detection and host resolution

## Register

```json
{
  "email": "user@example.com",
  "password": "Test1234",
  "username": "user123",
  "firstName": "Test",
  "lastName": "User"
}
```

Successful response:

```json
{
  "message": "User registered successfully"
}
```

## Refresh token

Request body:

```json
{
  "jwtToken": "expired-or-current-access-token"
}
```

Success requirements:

- the `refreshToken` cookie must exist
- the refresh token must belong to the user being refreshed
- if an access token is present and its subject does not match the refresh token subject, the request is rejected

## `GET /api/auth/me`

Typical response:

```json
{
  "user": {
    "id": "identity-user-id",
    "email": "user@example.com",
    "username": "user123",
    "firstName": "Test",
    "lastName": "User"
  },
  "roles": ["User"]
}
```

## Finance connection endpoints

These endpoints do not connect to a real banking provider from this service. They only maintain a connection flag in Redis.

| Method | Route | Auth | Effect |
| --- | --- | --- | --- |
| `GET` | `/api/auth/finance/status` | yes | returns whether the user is marked as connected |
| `POST` | `/api/auth/finance/connect` | yes | sets the user as connected |
| `DELETE` | `/api/auth/finance/disconnect` | yes | clears the user connection flag |

## Spotify OAuth endpoints

| Method | Route | Auth | Effect |
| --- | --- | --- | --- |
| `GET` | `/api/auth/spotify/authorize` | yes | returns the Spotify OAuth URL |
| `GET` | `/api/auth/spotify/callback` | no | exchanges `code` for tokens and redirects to the frontend |
| `GET` | `/api/auth/spotify/status` | yes | returns whether Spotify tokens exist in Redis |
| `DELETE` | `/api/auth/spotify/disconnect` | yes | removes Spotify tokens |

## Security and implementation notes

- Access tokens include `sub`, `email`, role claims, and user profile claims.
- Refresh tokens are JWTs signed with a separate secret.
- In development, `IdentityService` applies migrations on startup and seeds default roles.

<Callout type="warning" title="Access token lifetime mismatch">
  The configuration key is named `JwtSettings:AccessTokenExpirationMinutes`, but the current `AuthService` implementation uses `AddDays(...)` when building the access token expiry. That means the effective lifetime is longer than the configuration name suggests until the code is corrected.
</Callout>
