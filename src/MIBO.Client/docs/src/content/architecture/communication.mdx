# Communication

MIBO uses a small set of active communication patterns across the client, gateway, services, and AI runtime.

## 1. Client to platform: HTTP plus SSE

The main frontend runtime uses standard HTTP requests for CRUD and action execution, and SSE for chat streaming.

| Flow | Transport | Current status |
| --- | --- | --- |
| Login, register, refresh, `me` | HTTP | active |
| Conversation CRUD | HTTP | active |
| Chat request/response | HTTP request + SSE response | active |
| Generative UI data refresh | HTTP | active |
| Status dashboard | HTTP | active |

## 2. Gateway to downstream services

`MIBO.ApiGateway` authenticates the user and forwards requests to internal services.

### Identity propagation

The gateway adds:

- `X-User-Id` from the `sub` claim
- `X-User-Email` from the `email` claim

These headers are consumed downstream by `ConversationService` and `ActionService`.

## 3. Conversation service to AI agent

The most important internal flow is:

```text
Client -> Gateway -> ConversationService -> LangChainService
```

`ConversationService` sends:

- `session_id`
- `message`
- `user_id`

to `POST /chat` on `LangChainService`, then relays the response stream back to the browser.

## 4. AI agent to Action Service

The Python agent does not call external providers directly for product, finance, weather, or Spotify data. It invokes tool wrappers that call `ActionService`, which then dispatches to the correct handler.

This gives the platform:

- one shared action/query contract
- centralized field-hint inference
- shared retries and monitoring
- a single runtime for both agent tools and UI-triggered actions

## 5. Monitoring and retry flow

`ConversationService` records synthetic monitoring updates for:

- `platform`
- `ai-agent`

`ActionService` records monitoring and audit data for external integrations and can queue retries through RabbitMQ when retry mode is enabled.
