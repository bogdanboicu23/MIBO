# Architecture Overview

MIBO is a service-oriented platform with a clear active runtime path across the client, gateway, conversation runtime, action runtime, and AI pipeline.

Today, the core user-facing flow is:

1. the React client calls the **Ocelot** gateway
2. the gateway validates JWTs and forwards user identity via headers
3. `ConversationService` persists the user message and opens an SSE session to `LangChainService`
4. `LangChainService` plans and executes tools through `ActionService`
5. the final structured response is streamed back to the client as SSE
6. `ConversationService` persists the assistant payload in MongoDB

## Current runtime topology

```text
Browser / React Client
        |
        | HTTPS
        v
MIBO.ApiGateway (Ocelot)
        |
        +--> MIBO.IdentityService
        |
        +--> MIBO.ConversationService
        |         |
        |         +--> MongoDB (conversations, messages, UI instances)
        |         |
        |         +--> MIBO.LangChainService (HTTP + SSE)
        |                   |
        |                   +--> MIBO.ActionService
        |                               |
        |                               +--> BankService
        |                               +--> DummyJSON
        |                               +--> OpenWeatherMap
        |                               +--> Spotify API
        |                               +--> local Pomodoro config
        |
        +--> MIBO.ActionService

Supporting infrastructure:
- PostgreSQL for IdentityService
- Redis for finance connection state and Spotify tokens
- RabbitMQ for ActionService retry flow
```

## Key architectural characteristics

| Characteristic | Current state |
| --- | --- |
| Gateway | Ocelot |
| Main chat transport | HTTP plus SSE |
| Main AI orchestration | LangGraph pipeline in Python |
| Tool/data runtime | Action Service with typed handler routing |
| Primary persistence | PostgreSQL, MongoDB, Redis |
| Monitoring path | Mongo-backed integration status/audit data plus status frontend |

## Design choices that matter

- The client does **not** talk directly to the Python agent.
- The Python agent does **not** talk directly to external providers for the main data domains; it goes through `ActionService`.
- The main response path is synchronous streaming over HTTP plus SSE.
- The generative UI contract is persisted as assistant payload, then materialized into `ui.v1` on the client.
