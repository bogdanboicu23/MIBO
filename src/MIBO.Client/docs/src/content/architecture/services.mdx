# Service Descriptions

MIBO is composed of five backend services and a shared library. Each service is a standalone application with its own responsibilities, data sources, and deployment lifecycle.

<SectionDivider />

<SectionHeader title="Identity Service" description="Authentication, authorization, and user management" />

The Identity Service handles all authentication and user management concerns. It issues JWT tokens and manages the full user lifecycle.

  />
  <FeatureCard
    title="JWT Tokens"
    description="Access token generation and refresh token management"
    icon="🎟️"
  />
  <FeatureCard
    title="User Profiles"
    description="Profile storage, updates, and password management"
    icon="👤"
  />
</FeatureGrid>

<Callout type="info" title="Tech Stack">
  ASP.NET Core, Entity Framework Core, PostgreSQL, JWT Bearer Authentication
</Callout>

<SectionDivider />

<SectionHeader title="Conversation Service" description="AI chat orchestration and real-time streaming" />

The Conversation Service orchestrates chat interactions between users and the AI assistant. It integrates with LLM providers, manages conversation history, and streams responses in real time.

  <FeatureCard
    title="Chat Processing"
    description="Receives and processes user chat messages through the AI pipeline"
    icon="💬"
  />
  <FeatureCard
    title="AI Orchestration"
    description="Integrates with LLM providers to generate intelligent responses"
    icon="🤖"
  />
  <FeatureCard
    title="SSE Streaming"
    description="Streams AI response tokens to the client in real time"
    icon="🌊"
  />
  <FeatureCard
    title="Conversation History"
    description="Persists and retrieves past conversations for context"
    icon="📜"
  />
  <FeatureCard
    title="Action Execution"
    description="Triggers financial operations based on chat intent"
    icon="⚙️"
  />
  <FeatureCard
    title="Generative UI"
    description="Resolves dynamic UI components to render in the chat interface"
    icon="🎨"
  />
</FeatureGrid>

<Callout type="info" title="Tech Stack">
  ASP.NET Core, SignalR, AI/LLM Integration, NATS
</Callout>

<SectionDivider />

<SectionHeader title="Action Service" description="Reusable external data orchestration for live generative UI" />

The Action Service is the reusable external-data layer used by the generative UI runtime. It exposes normalized handlers that the AI agent and the frontend can call through the API Gateway for live data queries, refreshes, filters, and post-render interactions.

<FeatureGrid columns={3}>
  <FeatureCard
    title="Unified Query Contract"
    description="Exposes external integrations behind stable handler names such as products.catalog.query or crypto.coins.markets."
    icon="🧩"
  />
  <FeatureCard
    title="DummyJSON + CoinGecko"
    description="Aggregates product catalog data and crypto market data behind the same JSON contract."
    icon="🌐"
  />
  <FeatureCard
    title="Field Hints"
    description="Returns semantic metadata so generic tables, KPIs, charts, and detail cards can render deterministically."
    icon="🗂️"
  />
  <FeatureCard
    title="Transform Pipeline"
    description="Supports reusable row projections and computed fields before data reaches the frontend."
    icon="🧮"
  />
  <FeatureCard
    title="Refresh-Friendly"
    description="Designed for live reloads, search, pagination, sorting, and reopening conversations without regenerating the entire answer."
    icon="🔄"
  />
  <FeatureCard
    title="Service Decoupling"
    description="Keeps service-specific APIs out of the client and out of UI components."
    icon="🪢"
  />
</FeatureGrid>

<Callout type="info" title="Tech Stack">
  ASP.NET Core, HttpClient, CoinGecko Demo API v3, DummyJSON
</Callout>

The `MIBO.Shared` project provides common code that is referenced by multiple services, ensuring consistency and reducing duplication.

<FeatureGrid columns={2}>
  <FeatureCard

  <FeatureCard

    icon="🧰"
  />
  <FeatureCard
    title="Middleware"
    description="Shared middleware components for logging, error handling, and more"
    icon="🔧"
  />

<Callout type="tip" title="Why a Shared Library?">
  The shared library ensures that message contracts and DTOs stay in sync across services. Changes to a contract are caught at compile time rather than at runtime.
</Callout>
